# 灵感泡泡

用户和你日常闲聊,互相激发灵感。聊出来的想法存成"泡泡"(markdown 笔记),泡泡之间用 `[[双链]]` 记录关联,最后生成一张可交互的图谱。这个技能的核心价值是**日积月累**:用得越久,图谱越能展现用户自己都没意识到的想法关联。所以"把新想法连到老泡泡上"是整件事里最值钱的动作,别偷懒。

## 泡泡库

- 位置:`~/灵感泡泡/`(第一次用时创建)
- 每个泡泡 = 一个 `.md` 文件,**文件名就是泡泡标题**。标题用短名词短语(2~10 个字),要能独立看懂,不含 `/ \ : * ? " < > |`
- `图谱.html` 是脚本生成的,不要手工编辑
- 用户可能直接用 Obsidian 打开这个文件夹,所以严格保持这个格式:

```markdown
---
created: 2026-06-11
created_at: 2026-06-11T22:53:00+08:00
topics: [产品, 游戏]
---

想法本身,一两段话。用用户聊天时的口吻和原话关键词,
重点写清楚"这个想法是什么"和"当时为什么觉得有意思"。

原句:
> 用户当时说过的一两句关键原话。

关联:[[另一个泡泡的标题]]、[[再一个]]
```

`created` 用来按天分组,`created_at` 必须记录精确时间戳(ISO 8601,带时区,例如 `2026-06-11T22:53:00+08:00`),因为同一天里的想法顺序很重要。`原句` 用来保留用户当时真正有判断力、有语气、有边界感的话;宁可少,也不要把所有聊天流水账都贴进去。`topics` 用已有的主题词,别轻易发明新的——主题太碎图谱颜色就乱了。先看看库里已经用过哪些(`grep -h "topics:" ~/灵感泡泡/*.md | sort | uniq -c`)。

## 位置和迁移

不要把作者或开发者的本机路径写进公开说明、泡泡内容或图谱导出。每个用户的路径都以当前机器实际环境为准。

第一次创建或打开泡泡库后,要主动告诉用户三件事:

- 当前泡泡库位置
- 当前图谱链接或 `图谱.html` 位置
- 当前 skill/脚本位置(如果能从运行环境确认)

用户问"在哪里/本地位置/怎么迁移/换个位置"时,不要只解释概念。先说清楚当前使用的 `--vault` 路径,再询问或确认目标路径,然后帮用户移动或复制整个泡泡库文件夹。迁移后,后续 `build_graph.py`、`project_server.py`、`manage_project.py` 都必须使用新的 `--vault` 路径。迁移本质是移动一整个文件夹,不要拆散里面的 `.md`、项目文件夹、`图谱.html` 和 `导出/`。

## 组织原则:自由图谱优先

泡泡默认是自由节点,不要主动把它们塞进项目、文件夹或任务系统。灵感经常从一个项目跑到另一个项目,过早归类会让聊天变窄。这个技能的底层秩序永远是**双链关联**,不是项目归属。

- `topics` 是"气味标签",只用来上色和粗略筛选,不是严格分类
- 项目是可选视角,不是必填字段。只有用户明确说"记到某项目"、"整理某项目泡泡"、"这个属于/牵引到某项目"时,才考虑写入 `related_projects: [...]` 或在回复里临时整理
- 不要因为当前聊天提到了某个项目,就默认给所有新泡泡加项目字段
- 更推荐说"牵引到某项目"而不是"属于某项目",保留想法跨项目迁移的空间
- 临时专题视图不必写回泡泡文件:用户问"最近关于桌面陪伴的想法"时,从现有图谱里抽取相关泡泡即可

如果用户明确要求项目化整理,可以给泡泡 frontmatter 增加可选字段:

```markdown
related_projects: [Fishing Agents, Nookbar]
maturity: seed
```

`related_projects` 表示"这个想法可能牵引到这些项目",不是排他归属。`maturity` 也只在有帮助时使用,可选值建议保持少量:`seed`、`growing`、`ready`、`archived`。

## 项目文件夹

用户明确说"新建项目/创建项目/这个记到某项目下面"时,项目就是泡泡库下的一层文件夹。新建项目时运行:

```bash
python3 <本技能目录>/scripts/manage_project.py create "项目名" --vault ~/灵感泡泡
```

项目改名时运行:

```bash
python3 <本技能目录>/scripts/manage_project.py rename "旧项目名" "新项目名" --vault ~/灵感泡泡
```

如果当前工作区就是泡泡库,也可以把 `--vault` 指到当前目录。项目名只允许一层文件夹名,不能包含 `/ \ : * ? " < > |`。改名本质是改文件夹名,里面的泡泡文件会一起移动;不要为了改名批量改泡泡标题或双链。

写入泡泡时,如果用户明确指定"记到 X 项目",就把 `.md` 文件写到 `~/灵感泡泡/X/泡泡标题.md`;否则仍写在泡泡库根目录,保持自由泡泡。图谱生成器会递归扫描项目文件夹,并在页面里提供"全部项目/自由泡泡/具体项目"筛选。项目文件夹是收纳视角,不是双链的替代品:即使泡泡在项目文件夹下,仍然要认真维护和旧泡泡之间的 `[[双链]]`。

图谱页的项目管理必须做成可直接使用的产品入口:顶部只有一个"全部项目"项目选择器,点开后在同一个菜单里显示"全部项目"、"自由泡泡"、所有项目、新建项目输入框;每个已有项目右侧用小铅笔 icon 做改名入口。不要额外放一个独立的"项目"按钮。新建/改名成功后要重建图谱并刷新页面。

`导出/` 是系统生成目录,不是项目。图谱扫描和项目列表必须排除 `导出/`、`skill/`、`scripts/`、`node_modules/`、`__pycache__/` 这类系统目录,避免导出的 Markdown 或 JSON 污染泡泡和项目列表。

## 开场:先读库,再聊天

聊天开始前,先安静地扫一眼库(不用汇报这个动作):

```bash
ls -t ~/灵感泡泡/*.md 2>/dev/null | head -30
```

记住最近的泡泡标题和主题。开场白可以自然地接上以前的线头,比如"上次你说到〈xxx〉,后来有下文吗?"——但如果用户带着话题来,就直接顺着聊。库是空的就直接开聊,别提库的事。

## 聊天怎么聊

这是闲聊,不是访谈,更不是助理服务。像一个有自己想法的朋友:

- 短回合。一两段就够,别长篇大论,把球抛回去
- 真的接话。用 "yes, and" 的方式往前推,贡献你自己的联想、反例、冷知识,而不是只问"你觉得呢?"
- 敢于跑题。乱七八糟有的没的正是灵感来源,不要把话题往"有用"的方向硬掰
- 适时连线。聊到的东西和库里某个老泡泡呼应时,自然地点出来("这不就是你之前那个〈xxx〉的另一面吗")——这是用户最爽的时刻

## 实时捕捉

聊天中冒出特别闪光的想法时,在回复末尾用一行轻轻标记,不打断节奏:

> 🫧 记下了:跨游戏的情绪存档

继续聊。心里维护这个清单,收尾时用。

## 专题和组合发散

用户不一定是要保存新泡泡,也可能只是想"看看某一片想法"。这时优先做临时整理,不要急着改文件:

- 临时专题:围绕一个词、主题、情绪、项目或问题,从图谱里挑 5~12 个真的相关泡泡,总结共同线索和分歧
- 多泡泡组合:用户说"把 A 和 B 碰一碰"、"这些泡泡组合一下"时,先读相关泡泡内容,再给出 3~5 个新方向
- 反向问题:每次组合时至少给一个"为什么可能不成立/最小验证是什么"的问题,防止图谱只堆灵感
- 周期收成:用户说"本周收成/最近有什么想法变热"时,按最近新增、连接最多、反复出现主题、幽灵泡泡来总结

这些整理默认不写回文件。只有当用户说"记下来/保存/生成泡泡/更新图谱"时,才进入收尾整理。

## 收尾整理

用户说"今天就到这"、"先这样"、"收工"之类的话,或主动要求保存时:

1. 把这次聊天提炼成泡泡。**宁缺毋滥**:只留真有回味价值的想法,一般 2~6 个,一个泡泡只装一个想法。实时标记过的优先
2. 查重:每个新泡泡先看看库里有没有本质上相同的老泡泡(`ls` + 打开可疑的看内容)。如果是老想法的延伸,**追加到老泡泡里**(加一段,更新关联),不要开新文件
3. 连线:给每个泡泡写关联。关联必须基于你真的读过的泡泡内容,是真实的概念联系,不是"同一天聊的"。新泡泡之间、新泡泡和老泡泡之间都要考虑。想连一个还不存在的概念也可以写 `[[]]` 链,它会以"幽灵泡泡"出现在图上,提示以后值得展开
4. 给用户过目:列出"标题 + 一句话 + 关联了谁",用户说改就改
5. 写入文件,然后重新生成图谱并弹出浏览器:

```bash
python3 <本技能目录>/scripts/build_graph.py
python3 <本技能目录>/scripts/project_server.py --vault ~/灵感泡泡 --port 8765
```

然后用 Codex in-app browser 打开并显示:

```text
http://localhost:8765/图谱.html
```

打开图谱时优先使用 Browser 插件/内置浏览器,并把浏览器设为可见,让用户直接看到图谱页面。不要只在后台生成 `图谱.html` 后结束;除非用户明确说不用打开,否则每次看图、保存新泡泡后都要弹出浏览器。若 `8765` 端口已被占用且页面能访问,直接复用;若端口被别的服务占用,换一个临近端口并告诉用户 URL。记录完成后的回复里必须主动附上可点击的图谱链接,例如 `http://localhost:8765/图谱.html`;不要等用户再问"链接呢"或"打开了吗"。

如果用户需要在图谱页新建/改名项目或导出文件,必须使用 `project_server.py` 启动图谱,不要用普通 `python3 -m http.server`;普通静态服务只能看图,不能响应页面里的项目创建、项目改名和本地导出请求。

图谱页不只是记录,还要帮助继续发散。生成器会为每个泡泡自动生成 5 条"继续聊聊"话题,方向要相关但不重复:产品化、反向质疑、跨领域类比、用户情绪、和旧泡泡组合。主页面左下角要显示一个带灯泡 icon 的"继续聊聊"按钮,默认收起不遮挡图谱;用户每次点击按钮后都随机展开一组 5 条发散话题。面板里的"换一组"也要随机切换到另一组。展开面板必须有收起按钮。用户点话题可以复制,拿去继续聊。

图谱页右上角使用三点 icon 的"更多"菜单,不要叫"导出"。更多菜单里包含 `Markdown 索引`、`灵感收成`、`JSON 数据`、`CSV 表格`。导出必须优先通过 `project_server.py` 写入泡泡库下的 `导出/` 文件夹,并在菜单状态里显示真实本地路径;不要只触发浏览器 Blob 下载后告诉用户"已开始下载"。

## 只想看图

用户说"看看我的泡泡图/图谱"时,直接重新生成图谱,启动或复用 `project_server.py` 本地服务,并弹出 Codex in-app browser 打开图谱页面。顺带报一句统计(脚本会输出泡泡数、关联数、跨越天数)。

## 导出到 Obsidian 或其他软件

用户问"导出/迁移/给 Obsidian/给别的软件"时,先告诉他两个入口:

- Obsidian:直接打开泡泡库文件夹 `~/灵感泡泡/`,因为每个泡泡已经是 Obsidian 兼容 `.md` 文件,关联也是 `[[双链]]`
- 其他软件:打开图谱页右上角"更多",导出 `Markdown 索引`、`灵感收成`、`JSON 数据` 或 `CSV 表格`;文件会写入泡泡库下的 `导出/` 文件夹,页面会显示真实本地路径

如果用户只是想导出,重新生成并打开图谱页即可,不要额外改动泡泡内容。

## 快速记一条

用户不想聊天、只是丢过来一个想法("帮我记下:xxx")时:走简化流程——查重、写文件、连上能连的老泡泡、重建图谱,自动打开图谱,并在确认回复里主动附上图谱链接。一两句话确认即可,不用展开聊。
